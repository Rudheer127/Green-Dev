================================================================================

VISUAL:

SPEAKER SCRIPT (45 seconds):

"Here's the problem: Student developers build for speed and functionality. 
They don't think about cloud sustainability because they literally can't see it.

Consider this: An always-on EC2 instance runs 720 hours per month. But a 
typical student project—it's actually used maybe 50 hours per month. That 
means 670 hours of wasted compute every single month. Wasted electricity. 
Wasted money.

That's a 14× cost inefficiency for a single architectural choice.

And it's not just compute. Missing dependency caching means 40% of every 
CI run is just re-downloading the same npm packages. Oversized Docker base 
images mean 1.1GB transfers instead of 50MB. These inefficiencies are 
everywhere in student projects.

But here's the thing: Most developers have NO IDEA any of this is happening. 
There's no visibility. No feedback loop. No way to learn."

SLIDE 5: HOW IT WORKS - "Smart, Not Wasteful"
================================================================================

VISUAL:
├─ System architecture diagram (simplified):
│  ├─ Frontend: Next.js + Vercel
│  ├─ Analysis: 5 deterministic engines
│  ├─ AI: AWS Bedrock (Claude Sonnet)
│  ├─ Database: Supabase + pgvector cache
│  └─ Flow: All serverless, scales to zero
├─ Key stat: "pgvector semantic cache: Avoid redundant LLM calls"

SPEAKER SCRIPT (50 seconds):

"Here's what makes us different from pure LLM tools: We're deterministic.

We don't send your entire codebase to Claude and ask 'what's wrong?' That's 
wasteful and unreproducible. Instead, we use five specialized rule engines— 
each focused on one dimension of sustainability—to extract structured findings.

Those findings are checked against our pgvector semantic cache in Supabase. 
If we've seen a similar pattern before, we use the cached explanation. 
No LLM invocation. No tokens. No latency. No CO2.

Only on a cache miss do we invoke Claude with the structured findings JSON. 
Claude then synthesizes those into four tailored report narratives.

This approach is:
- Fast: Rules execute in milliseconds
- Consistent: Same repo = same findings
- Explainable: We can tell you exactly why we flagged something
- Efficient: Our infrastructure itself is sustainable

The whole thing runs serverless: Vercel for the frontend, Lambda for analysis, 
Bedrock for AI, Supabase that auto-scales. When no one's using it, the bill 
is essentially zero. Sustainability at the architecture level."

Q&A TALKING POINTS
================================================================================

Q: "How do you handle accuracy for projects you've never seen?"
A: "Great question. We use deterministic rules that are derived from empirical 
data. For example, we know that node:latest is ~1.1GB and node:alpine is ~50MB. 
That's not an estimate. It's a fact. Same with CI patterns—we're flagging proven 
inefficiencies that match open-source best practices. So accuracy is high for 
'unknown' repos because we're not guessing; we're matching against facts."

Q: "What about false positives?"
A: "We're conservative. We flag something as an issue only if we're quite 
confident it IS an issue. A few false positives is acceptable because the 
user can dismiss them. The cost of a missed opportunity (not flagging something 
fixable) is higher than the cost of a false positive."

Q: "How does this scale?"
A: "Our entire stack is serverless. Frontend on Vercel Edge (scales globally), 
API routes on Vercel Functions (auto-scale), analysis on AWS Lambda (pay-per-
invocation), database on Supabase (auto-scaling managed Postgres). The only 
constraint is GitHub API rate limits (60/hour unauthenticated, 5000/hour with 
token). At 10k scans/month, we're well within limits."

Q: "Why does this benefit AWS/Vercel/companies?"
A: "This is fundamentally educational software. We're teaching developers best 
practices that benefit AWS (happy customers who build well), benefit developers 
(better architectures), and benefit the planet. The fact that we run on AWS is 
a natural fit—we're the 'sustainability layer' on top of their infrastructure."

Q: "How do you prevent misuse (too-aggressive recommendations)?"
A: "All recommendations go through a human review process before shipping. We've 
built in a principle: never recommend something that has a trade-off. Every 
suggestion benefits the dev in speed, cost, or quality. That self-polices."

Q: "What's your competitive advantage?"
A: "Three things: First, we're zero-friction (no billing integration required). 
Second, we're student-focused (language, UX, pricing). Third, we're code-centric 
(not just bills). Existing enterprise tools require deep billing access and 
provide raw metrics. We provide specific, implementable recommendations."

Q: "What's your long-term business model?"
A: "Freemium SaaS plus partnerships. Free tier keeps everyone engaged. Pro tier 
($9/mo) for power users. Enterprise serves teams and companies. University 
partnerships provide stable licensing revenue. None of these are cannibalizing; 
they're complementary segments."

Q: "Can you open-source this?"
A: "That's the plan for v2. The analysis engines are standalone libraries 
(Python packages, npm modules). Community can build on them. We open-source, 
they use locally or self-host. We stay profitable on the cloud SaaS offering, 
paid partnerships, and enterprise tier. Open source doesn't cannibalize us; 
it extends our reach."

SLIDE DECK VISUAL STYLE GUIDE
================================================================================

COLOR PALETTE (Use Consistently):
├─ Dark background: #171614 (every slide)
├─ Text: White / light gray
├─ Primary accent: Teal #4f98a3 (calls-to-action, key metrics)
├─ Secondary accent: Gold #e8af34 (medium-priority items)
├─ High-priority/alert: Red #dc2626 (issues, problems)
└─ Success: Green #22c55e (wins, improvements)

TYPOGRAPHY:
├─ Headings: Cabinet Grotesk (bold, 700-800 weight)
├─ Body text: Geist (regular, 400 weight)
├─ Code/Numbers: Monospace (for technical credibility)
└─ Slide headings: 48-56px, body 24-32px

LAYOUT PRINCIPLES:
├─ Dense but not crowded (whitespace is premium)
├─ One main idea per slide
├─ Data visualizations > text (use icons, charts, infographics)
├─ Consistent spacing (8px/16px/24px grid)
└─ Animations: Smooth transitions, 300-500ms (not distracting)

ANIMATIONS (Use Sparingly):
├─ Slide transitions: Fade in (300ms)
├─ Statistics: Count-up animation (2 seconds)
├─ Charts: Draw in (1 second)
├─ Avoid: Spinning wheels, sound effects, distracting transitions
